## 1. 큰 뼈대

**① 설계 루프** (증명을 통과해야 다음 단계로)

```text
문제 상황 인식 → 문제 상황 구조화 → 해법 구상 → 해법의 정당성 증명
         ↑______________________________________________|
         (증명이 안 되면 다시 위로)
```

**② 구현 루프**

```text
구현 → 디버깅
 ↑______|
 (버그 나면 다시 구현)
```

---

## 2. 각 단계에서 실제로 하는 일

| 단계 | 하는 일 |
| --- | --- |
| **문제 재정의** | 원래 문제를 "정답 후보가 무엇인가"로 다시 쓴다 |
| **구조화** | 데이터를 어떤 형태로 들고 있을지 정한다 (배열, 카운팅 배열, 해시셋 등) |
| **수도코드화** | 실제 코드가 아닌 의사코드로 로직을 적는다 |
| **정당성 증명** | 이 로직이 항상 정답을 내는 이유를 설명한다 |
| **병목 관찰** | 시간·공간복잡도를 계산해 어디가 느린지 손가락으로 짚는다 |
| **비효율 관찰 → 재시작** | 병목을 없앨 아이디어를 관찰하고 사이클을 다시 돈다 |

## 3. 예제로 사이클 돌려보기

> **문제:** 정수 배열 `arr`(크기 N)와 목표값 `K`가 주어질 때, `arr[i] + arr[j] = K`인 쌍이 있으면 YES, 없으면 NO.

### 1바퀴 — Naive (완전탐색)

- **구조화:** 배열 그대로 두고, 이중 반복문으로 모든 쌍을 확인한다.

```text
for i: 0 ~ N-1
    for j: i+1 ~ N-1
        if arr[i] + arr[j] == K
            return YES
return NO
```

- **증명:** 모든 쌍을 빠짐없이 검사하므로 답이 존재하면 반드시 찾는다.
- **병목:** 시간복잡도 `O(N²)`. N = 10만이면 100억 연산 → 시간초과.
- *"굳이 모든 쌍을 다 봐야 하나?"* 라는 비효율을 관찰하고 다시 구조화로 올라간다.

### 2바퀴 — 정렬 + 투 포인터

- **관찰:** `arr[i]`를 하나 정하면 찾는 짝은 `K - arr[i]` 딱 하나뿐이다.
- **구조화:** 배열을 **정렬**해두면 양 끝 포인터로 좁혀갈 수 있다.

```text
정렬(arr)
left = 0, right = N-1
while left < right
    sum = arr[left] + arr[right]
    if sum == K  → return YES
    if sum < K   → left++      (더 큰 값 필요)
    if sum > K   → right--     (더 작은 값 필요)
return NO
```

- **증명:** sum이 작으면 왼쪽↑, 크면 오른쪽↓. 이 단조성 덕분에 놓치는 쌍이 없다 ✅
- **병목:** 정렬 `O(N log N)` + 투 포인터 `O(N)` = `O(N log N)`. 이제 정렬이 병목이다.

### 3바퀴 — 해시셋

- **관찰:** 정렬조차 필요 없다. `arr[i]`를 볼 때 `K - arr[i]`가 **이미 지나온 수 중에 있었나**만 알면 된다.
- **구조화:** 지나온 수를 **해시셋**에 저장한다 (조회 O(1)).

```text
seen = 빈 셋
for x in arr
    if (K - x) in seen
        return YES
    seen.add(x)
return NO
```

- **증명:** x를 보는 시점에 seen에는 앞쪽 원소가 전부 들어 있다. 짝이 있으면 그 쌍의 뒤쪽 원소에서 반드시 걸린다 ✅
- **복잡도:** 시간 `O(N)`, 공간 `O(N)`. 시간 병목이 사라졌다. 끝.

## 4. 개선 흐름 정리

```text
O(N²)  →  O(N log N)  →  O(N)
Naive     정렬+투포인터     해시셋
```
